# itukara - Step 3.5: Repository Interface Design

このドキュメントでは、Step 3で設計した各Entityに対応する、"Domain 層の Repository インターフェース（`abstract class`）" の設計を行います。 
実装（Data層のSQLite用具象クラス等）はここには記載せず、アプリケーションが「何を必要としているか（要求仕様）」の定義に徹します。

## 1. 非同期設計方針（Future / Stream）

Riverpod での `AsyncValue` サポートとの親和性を高めるため、以下の原則で設計します。
- "読み取り（単発）/ 書き込み・更新処理": `Future` を用いる。
  - 例: `Future<void> saveRule(...)`, `Future<UserEntity> getUser(...)`
- "リアルタイム監視（リストや状態の変更検知）": `Stream` を用いる。
  - 例: `Stream<List<TaskEntity>> watchTasks()`
  - ローカル(SQLite等)の場合は、DB更新時に手動でStreamへイベントを流す仕組み（RxDart等）を用いるか、Riverpod自体でステートを管理しても良いですが、Firebase(Firestore)移行時には `.snapshots()` を垂れ流すだけで UI が自動更新される強力なメリットがあります。

## 2. 各Repositoryの定義

### UserRepository
- "責務": アプリを利用するユーザー自身を表すデータモデルの永続化および取得。
- "将来拡張": 認証・同期フロー（Firebase Auth）が入る際に、ログイン中ユーザー情報の提供元となります。

> **Note**: 実装コードは `frontend/app/lib/features/auth/domain/user_repository.dart` に抽出済みです。

### PremiumRepository
- "責務": アプリ内課金（IAP）やそれに伴うサーバーサイドの検証結果（PremiumStatus）の取得と管理。
- "将来拡張": 課金復元や、FastAPI側でのレシート検証結果を同期する役割を持ちます。

> **Note**: 実装コードは `frontend/app/lib/features/premium/domain/premium_repository.dart` に抽出済みです。

### TaskRepository
- "責務": ユーザーが定義するタスクの親（カテゴリ）メタデータの永続化。
- "将来拡張": `TaskRecordRepository`と独立していますが、実装(Data Layer)においては `TaskRecord` 保存時に `lastRecordedAt` を一貫して更新する責任を持ちえます。

> **Note**: 実装コードは `frontend/app/lib/features/task/domain/task_repository.dart` に抽出済みです。

### TaskRecordRepository
- "責務": タスク実行履歴の永続化、および AI 連携（異常検知）などの統計分析に必要な複雑なデータクエリの提供。
- "一覧表示高速化戦略": 「一覧画面の最後にやった時間は？」というクエリは発行せず、 `TaskEntity.lastRecordedAt` を参照します。この Repository は純粋にグラフ表示や履歴一覧、AIの時系列データ抽出の目的に特化します。

> **Note**: 実装コードは `frontend/app/lib/features/task/domain/task_record_repository.dart` に抽出済みです。
